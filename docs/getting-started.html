Getting started with Gither

Use Gither as the code-forge layer around an existing Git checkout: snapshot the repo, record why a change exists, run the local gate, and publish a review packet before a hosted mirror receives the merge.

First run Change flow Public URLs

First run

Start in the repository that should be reviewed. Gither does not replace Git objects yet; it adds review state, ownership state, and release discipline around them.

python -m pip install -e .
gither repo-snapshot --repo .
gither python-audit --root src
gither gate --repo . --python-root src
A clean gate means the current checkout is understandable enough to review. It is not a release by itself.

Change flow

  1. 1. Route the work Pick the repository and adjacent dependencies before editing. 
    gither route "add a docs wiki for Gither"
  2. 2. Capture the starting state Keep branch, head commit, remote, and dirty status visible. 
    gither repo-snapshot --repo .
  3. 3. Write a change note The note records why the change exists and which tests prove it. 
    gither change-note \
  --summary "add Gither wiki pages" \
  --why "operators need public instructions before using Gither as a forge" \
  --test "python -m pytest -q"
  4. 4. Run the local review gate Gate the repo before asking another human or agent to review. 
    gither gate --repo . --python-root src --strict
  5. 5. Build a portable review packet The packet is the handoff artifact for PR review or p2p review. 
    gither review-pack --repo . --python-root src --output artifacts/review-pack.md
  6. 6. Publish to mirrors GitHub Pages, GitHub PRs, and 5mart are mirrors. Gither should keep the authority record.

When to use each command

Command Use it when Output
repo-snapshot You need the exact Git state before review. Branch, commit, remotes, dirty files.
change-note A change needs durable context. A versioned JSON note under .gither/changes.
python-audit Python symbols need annotation and docstring discipline. A quality report.
gate You are ready to prove a checkout is reviewable. Pass or fail gate status.
review-pack A reviewer needs a portable summary. Markdown review packet.
p2p-manifest A workspace needs to travel without trusting the transport. Content-addressed repo manifest.
