Repository gitbooks

Every repository in the Knitweb workspace should have a living gitbook: quick enough for first contact, deep enough for operators, and tied to merged PR evidence through Gither.

Book standard

A Gither gitbook is not a marketing page. It is the operator surface for a repository.

Start

Project purpose, install path, first command, supported mirrors, and current owner.

Concepts

Stable domain model, trust assumptions, data records, and how the repo fits the Knitweb graph.

Guides

Task recipes for maintainers, reviewers, release operators, and agentic contributors.

Reference

Commands, APIs, schemas, config, ports, environment variables, and compatibility notes.

Evidence

Tests, review gates, merged PRs, enrichment score exports, release receipts, and known limitations.

Recovery

Mirror failures, dirty worktrees, dependency drift, key rotation, rollback, and incident playbooks.

Research baseline

The structure is informed by documentation systems that separate concepts, guides, reference material, and operational examples.

  • Quip docs for compact GitBook-style onboarding and navigation.
  • libp2p docs for peer-to-peer concepts, protocols, and implementation references.
  • IPFS docs for content-addressed concepts, how-to pages, and reference material.
  • Radicle docs for sovereign code collaboration and peer-to-peer forge operations.
  • IBM Quantum docs and PennyLane docs for quantum tutorials, API references, and learning paths.
