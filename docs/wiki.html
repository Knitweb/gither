Wiki for the Gither code forge

This wiki is the operating manual for the current Gither system: what the records mean, which commands create them, and how a change moves from local checkout to reviewed mirror.

Core records

Record Meaning Current implementation
Repository snapshot Observable Git state for a checkout. gither repo-snapshot --repo .
Change note Why a change exists, how it was tested, and what a reviewer must know. .gither/changes/*.json
Review gate The local acceptance check before a mirror receives work. gither gate --repo . --python-root src
Review pack Portable reviewer handoff with repo state and audit evidence. gither review-pack
RID Content-addressed repository identity. gither.identity
Signed refs Peer-verifiable publication of refs. gither.sigrefs
P2P manifest Workspace state that can be verified without trusting GitHub or GitLab. gither p2p-manifest
Ownership model Usage-based royalty policy for reviewed components. gither value-model
Contributor enrichment score Contributor signal based on merged PR evidence, review state, tests, docs, dependency surfaces, and optional Pulse usage receipts. gither enrichment-score --input artifacts/merged-prs.json
Repository gitbook Living operator documentation for each Knitweb repository. docs/repository-gitbooks.html

Command map

Inspect

repo-snapshot, discover, and graph map the checkout and workspace before edits start.

Explain

change-note turns developer intent into a durable, reviewable record.

Gate

python-audit and gate make code quality explicit before a PR or release.

Package

review-pack and p2p-manifest create portable handoff artifacts.

Verify

p2p-verify checks received repository state without trusting the transport.

Value

value-model and license-protocol define ownership and mirror constraints.

Enrichment

enrichment-score turns merged PR exports into a contributor signal without rewarding raw commit volume.

Decision rules

  • Use Git for the object model and transport while Gither is still early.
  • Treat GitHub and GitLab as mirrors, not the long-term source of authority.
  • Do not reward raw commit volume. Code earns only when it is reviewed, released, depended on, and actually used.
  • Use merged PRs as the tracking unit for contributor enrichment, then require review, gate, test, documentation, dependency, or usage evidence for stronger signals.
  • Keep documentation tied to code changes. If a page does not help an operator review, release, or recover the system, it should be shorter.
  • Prefer local gates and portable artifacts before hosted automation.
  • Mirror to public sites only after the repo state and docs links have been tested.

Troubleshooting

Symptom Likely cause Action
Gate fails on dirty state Untracked or modified files are still present. Inspect with git status --short, commit intended files, and ignore or remove generated cache files.
Python audit fails A public symbol lacks annotations, docstrings, or manageable size. Fix the code before documenting around the failure.
Public page is 404 The GitHub Pages mirror or 5mart mirror has not received the new static files. Publish the repo PR, then mirror the docs/ output to the public web root.
Reviewer cannot reconstruct the work No change note or review pack was produced. Run gither change-note and gither review-pack before review.
