Gither enrichment score

The enrichment score ranks contributors who improve other people's codebases through merged, reviewed pull requests. It is a contribution signal, not a direct payment rule.

Score model Merged PR tracking Command Publication

Score model

Only accepted work counts. Drafts, open PRs, closed-unmerged PRs, raw commits, and generated volume do not produce enrichment points.

Evidence Points Reason
Merged PR 40 The change entered the target branch and can be audited later.
Reviewer evidence 6 per review, capped at 4 Reviewed work is harder to farm than raw PR volume.
Gate passed 18 Gither should reward accepted quality gates before mirror publication.
Tests touched 12 Tests make the enriched codebase safer for the next maintainer.
Docs touched 8 Operational knowledge is part of the codebase.
Dependency or integration surface touched 14 Shared surfaces enrich downstream users when they are handled carefully.
Usage receipts 4 per receipt, capped at 12 Pulse usage evidence can raise confidence that the contribution matters in practice.
Bounded footprint Small capped bonus Changed files and churn are capped so line-count farming stays weak.

Merged PR tracking

  1. Export PR evidence. Use GitHub, GitLab, or a Gither mirror export that contains author, state, merge time, reviews, checks, labels, files, and optional Pulse usage receipts.
  2. Normalize locally. Gither converts GitHub-like JSON into stable pull-request evidence records.
  3. Score only merged work. The calculator drops any record that is not merged.
  4. Publish the leaderboard as evidence. The score can be linked from the repository gitbook and the public Gither wiki.

Command

Store a PR export under an artifact path, then score contributors offline.

gither enrichment-score --input artifacts/merged-prs.json
gither enrichment-score --input artifacts/merged-prs.json --json
The command accepts a JSON list or an object with pull_requests, pullRequests, items, or nodes.
